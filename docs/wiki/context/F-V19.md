## Restrições da tarefa

### Anti-SPEC aplicável

- **§1** `members.sponsor_id` — **adicionar apenas** (lead vira member novo). Nunca atualizar valor existente.
- **§2** `shopify_customers`/tags — **adicionar apenas** tag `subscriber:<YYYY-MM-DD>`. Não remover tags existentes (ex.: `cliente:vip`).
- **§4** Webhook Shopify `orders/paid` — **NÃO TOCAR**. F-V03 continua funcionando. Hook F-V03 vira no-op via flag em `Onda 3` pós-MVP.
- **§6** Migrations idempotentes (`CREATE … IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`) com rollback comentado no topo.
- **§11** Provider Guru fica em `lib/subscriptions/providers/guru.ts` com interface agnóstica (espelha `lib/payouts/v2/`).

### Feature flag

- **`LRP_V2_GURU_FLOW`** (default `false` em prod). Quando `false`:
  - `/r/[slug]` lookup secundário em `members.ref_code` desabilita; rota responde só F-V15 eventos.
  - `/api/webhooks/guru` continua aceitando (não está atrás da flag), mas `markSubscriptionPaid` é o caminho v2 já gated.
  - Cron `inactivate-expired-subscriptions` respeita `CRON_DISABLED_V2`.
- Helper: adicionar `isV2GuruFlowEnabled()` em `lib/utils/featureFlags.ts`.

### Comportamento crítico

- **Idempotência por email + sponsor_id** no `createPreRegistration` — submit duplicado reusa member existente.
- **Idempotência por `X-Request-ID`** no webhook Guru — segundo dispatch idêntico não reprocessa.
- **Try/catch isolado** em cada caso do switch do webhook (Anti-SPEC §4 pattern) — falha em 1 evento NÃO derruba 200.
- **MVP de hoje:** Shopify sync **mock** (`SHOPIFY_SUBSCRIPTION_SYNC_LIVE=false` default) — apenas `console.info` + insert local em `orders`.

### 🟡 Gaps documentados no runbook (precisam confirmar logando no Guru)

(Ver `docs/wiki/runbooks/webhook-guru-debug.md` §"Gaps a confirmar logando" — 7 itens.)

Principais:
- Guru **NÃO usa HMAC** (autenticação via `api_token` no body). SPEC original assumiu HMAC. → **Schema Zod do `providers/guru.ts` precisa ser re-escrito** com o schema CORRIGIDO do runbook.
